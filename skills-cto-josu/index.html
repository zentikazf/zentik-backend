📖 El Manual Prohibido — Flujo 100% Senior de Principio a Fin

El orden exacto para llevar un proyecto de cero a produccion usando las 19 skills. Sin atajos.

Fase 1 — Fundacion

1

/senior-project-generator — Arquitectura antes que codigo. Sin blueprint no hay proyecto, hay improvisacion.

2

/project-setup-enforcer — Enforcement desde el dia 1. ESLint, Prettier, Husky, commitlint, CI. Si no bloqueas codigo malo en el commit, lo pagas en la auditoria.

Fase 2 — Construccion (por cada modulo del blueprint)

3

/senior-feature-builder x N — Cada modulo con mini-blueprint aprobado. Nunca codear directo.

Las skills de referencia se activan solas durante la implementacion:
nestjs-best-practices · next-best-practices · api-design · error-handling-patterns · brainstorming

Fase 3 — Auditoria pre-deploy (cuando el MVP esta completo)

4

/blueprint-compliance-auditor — Verificar que lo construido cumple lo disenado. Score 0-100%.

5

/senior-code-auditor — Calidad de codigo: arquitectura, nomenclatura, patrones, rendimiento.

6

/frontend-nextjs-auditor — RSC boundaries, error.tsx, loading.tsx, performance, middleware.

7

/prisma-schema-auditor — Indices, relaciones, soft delete, naming. ANTES de migrar en prod.

8

/security-hardening-auditor — Helmet, CORS, auth, cookies, CSRF, file uploads. No se deploya sin esto.

9

/testing-enforcer — Verificar que tests cubren los flujos criticos.

10

/deploy-readiness-checker — Ultimo checkpoint: env vars, CI/CD, hosting, Sentry, logging, build.

Fase 4 — Deploy

🚀

Deploy a produccion — Solo despues de pasar las 7 auditorias.

Fase 5 — Mantenimiento en produccion

11

/context-aware-coder — QA fixes y cambios menores. Los fixes siguen los patrones existentes, no rompen convenciones.

12

/senior-feature-builder — Features nuevos repiten Fase 2 + auditorias de lo que se toco.

13

Auditorias puntuales — Solo las que aplican: toque DB → /prisma-schema-auditor, toque auth → /security-hardening-auditor, etc.

Uso especial (no estan en el flujo diario)

*

/project-rescue — Solo para proyectos heredados/deteriorados · /copywriting — Textos de marketing · /skill-creator — Crear o mejorar skills

Fichas Tecnicas

Cada skill con su ficha completa: que hace, cuando usarla, que genera, tips y prompts de ejemplo.

senior-project-generator
Generator▼
📌 QUE HACE

Actua como un CTO senior que entrevista al usuario sobre su idea de proyecto y genera un PROJECT_BLUEPRINT.md de 1000+ lineas. Cubre arquitectura completa, schema de base de datos, endpoints por modulo, estrategia de autenticacion, CI/CD, deployment y checklist pre-produccion. Es el documento fundacional de todo proyecto nuevo.

⏰ CUANDO USAR
  • Cuando arrancas un proyecto de cero y no existe nada todavia
  • Cuando quieres disenar la arquitectura antes de escribir codigo
  • Cuando necesitas un documento de referencia para el equipo
📦 QUE GENERA
PROJECT_BLUEPRINT.md — Documento completo con: vision del proyecto, stack tecnologico, estructura de carpetas, schema Prisma detallado, endpoints REST por modulo con request/response tipados, configuracion de auth (JWT, guards, RBAC), CI/CD pipeline, variables de entorno, checklist de produccion.
💡 TIPS Y ERRORES COMUNES

✅ Se especifico con el dominio: "plataforma SaaS de gestion de proyectos multi-tenant"

✅ Responde todas las preguntas del cuestionario con detalle

❌ No digas "haceme un proyecto" sin contexto — el blueprint sale generico

❌ No saltees el cuestionario — los huecos te muerden despues

📝 PROMPTS DE EJEMPLO
"Crear nuevo proyecto de gestión de inventarios para una empresa de logística"
"Nuevo proyecto de e-commerce con multi-tenant y pasarela de pagos"
"Generar blueprint para plataforma SaaS de gestión de proyectos"
senior-feature-builder
Builder▼
📌 QUE HACE

Dirige el proceso completo de creacion de un feature nuevo: desde la idea vaga hasta el codigo implementado. Tiene 5 fases (Brainstorming, Discovery, Technical Design, Blueprint, Implementacion) con un hard gate que impide escribir codigo sin blueprint aprobado. Se adapta al proyecto existente leyendo la estructura real.

⏰ CUANDO USAR
  • Cuando necesitas agregar funcionalidad nueva y completa (modulo, feature, flujo)
  • Cuando tenes una idea y queres que alguien te guie desde el discovery
  • Cuando queres que el codigo nuevo siga la arquitectura del proyecto
📦 QUE GENERA
FEATURE_BLUEPRINT.md — Mini-PRD con: objetivo, endpoints con request/response, schema Prisma, estructura de archivos, plan de implementacion, checklist pre-merge.
Codigo implementado — DB schema, modulo NestJS completo (controller + service + DTOs), pagina Next.js con loading/error states.
💡 TIPS Y ERRORES COMUNES

✅ Si es solo backend, decilo en la Pregunta 3 para activar el skip inteligente

✅ Responde las preguntas una a la vez — la calidad del blueprint depende de tus respuestas

❌ No digas "implementa X rapido" — el hard gate te frena. Primero blueprint, despues codigo

❌ No digas "hace todo" — responde cada fase para obtener un resultado senior

📝 PROMPTS DE EJEMPLO
"Nuevo feature: sistema de notificaciones con push y email"
"Quiero agregar facturación con integración a Stripe"
"Nuevo módulo de reportes con exportación a PDF y Excel"
context-aware-coder
Builder▼
📌 QUE HACE

Antes de escribir una sola linea, escanea el proyecto entero para aprender los patrones establecidos: estructura de modulos, DTOs, stores, providers, API client, naming, error handling. Despues escribe codigo que se integra perfectamente con lo existente. Es como un senior que acaba de unirse al equipo y su primera regla es "leo como trabaja este equipo antes de tocar nada".

⏰ CUANDO USAR
  • Cambios pequenos post-auditoria: agregar un campo, ajustar un endpoint, fix de QA
  • Cuando el proyecto ya esta clean y queres mantener la consistencia
  • Onboarding: pedile "quiero entender el proyecto" y genera un resumen expandido de patrones
  • Cualquier cambio que no sea un feature completo nuevo
📦 QUE GENERA
Resumen de patrones — 5-10 puntos: como se organizan modulos, DTOs, errores, eventos, guards, stores, API client, convenciones de naming e imports.
Codigo consistente — Sigue exactamente los mismos patrones que descubrio. Si el proyecto usa AppException, usa AppException. Si usa Zustand, usa Zustand.
💡 TIPS Y ERRORES COMUNES

✅ Ideal para QA fixes, agregar campos, modificar endpoints menores

✅ Usalo en modo onboarding: "dame un resumen de patrones del proyecto"

❌ No lo uses para features nuevos completos — para eso usa /senior-feature-builder

⚠️ Si el proyecto tiene inconsistencias, sigue el patron del modulo mas reciente

📝 PROMPTS DE EJEMPLO
"Agregar campo 'priority' al módulo de tareas siguiendo los patrones"
"Fix de QA: el endpoint de listar usuarios no pagina correctamente"
"Quiero entender el proyecto — dame un resumen de patrones"
project-rescue
Rescue▼
📌 QUE HACE

Diagnostica el estado de un proyecto heredado o deteriorado ejecutando 5 auditorias internas (codigo, seguridad, DB, tests, deploy). Genera un plan de rescate priorizado por severidad y opcionalmente crea un PROJECT_BLUEPRINT.md retroactivo para documentar lo que existe.

⏰ CUANDO USAR
  • Cuando recibes un repositorio heredado de otro equipo
  • Cuando un proyecto se degradó y necesitas un plan para elevarlo
  • Cuando quieres un diagnostico completo antes de decidir si rescatar o reescribir
📦 QUE GENERA
Diagnostico completo + 5 auditorias internas + plan de rescate priorizado por severidad + PROJECT_BLUEPRINT.md retroactivo (opcional).
💡 TIPS Y ERRORES COMUNES

✅ Usalo como primer paso — el plan de rescate te dice exactamente que priorizar

❌ No lo uses para proyectos tuyos desde cero — para eso usa auditorias individuales

❌ No intentes arreglar todo a la vez — sigue la prioridad del plan

📝 PROMPTS DE EJEMPLO
"Rescatar este proyecto"
"Auditar este proyecto heredado y darme un plan de acción"
blueprint-compliance-auditor
Auditor▼
📌 QUE HACE

Lee el PROJECT_BLUEPRINT.md y verifica seccion por seccion que la implementacion real cumpla con lo disenado. Es el auditor maestro que orquesta la verificacion de compliance: estructura, endpoints, schema, seguridad, tests, deploy. Genera un score 0-100% de preparacion.

⏰ CUANDO USAR
  • Antes de deploy a produccion — verificar que todo lo planeado esta implementado
  • Cuando tenes un PROJECT_BLUEPRINT.md y queres saber cuanto se cumplio
  • Para una auditoria "one-shot" en lugar de correr cada auditor por separado
📦 QUE GENERA
Informe unificado seccion por seccion contra el blueprint. Score de preparacion 0-100% con hallazgos y correcciones sugeridas.
💡 TIPS Y ERRORES COMUNES

⚠️ Requiere PROJECT_BLUEPRINT.md — sin blueprint no puede funcionar

❌ No funciona con FEATURE_BLUEPRINT.md — solo lee el blueprint del proyecto

✅ Si no tenes blueprint, usa auditorias individuales en su lugar

📝 PROMPTS DE EJEMPLO
"Auditoría completa contra el blueprint"
"Verificar cumplimiento del blueprint"
senior-code-auditor
Auditor▼
📌 QUE HACE

Actua como un Staff Engineer revisando el codigo. Audita arquitectura, nomenclatura, estructura de carpetas, patrones de API, manejo de errores, rendimiento y logging. Genera un informe puntuado A-F por categoria con referencias archivo:linea y codigo de correccion concreto para cada hallazgo.

⏰ CUANDO USAR
  • Despues de implementar un feature completo
  • Revision post-merge para verificar calidad
  • Cuando queres saber si tu codigo esta a nivel senior
📦 QUE GENERA
AUDIT_REPORT.md — Informe puntuado A-F por categoria: arquitectura, nomenclatura, patrones, errores, rendimiento, logging. Cada hallazgo con archivo:linea, severidad, problema y codigo de correccion.
💡 TIPS Y ERRORES COMUNES

✅ Apunta a modulos especificos: "auditar el modulo de usuarios"

❌ No pidas "auditar todo" sin contexto — es mejor por modulo para profundidad

📝 PROMPTS DE EJEMPLO
"Revisar la calidad de mi código"
"Auditar calidad del módulo de tareas y el de proyectos"
frontend-nextjs-auditor
Auditor▼
📌 QUE HACE

Audita a fondo el codigo frontend Next.js (App Router). Verifica 7 categorias con peso ponderado: Server/Client Components (20%), Data Fetching y Estado (15%), Error Handling (15%), Seguridad Frontend (15%), Performance (15%), Convenciones (10%), Middleware y Rutas (10%). Genera un informe con nota A-F, hallazgos con archivo:linea y correcciones concretas.

⏰ CUANDO USAR
  • Cuando queres una auditoria profunda del frontend especificamente
  • Despues de implementar varias paginas y componentes
  • Antes de deploy — verificar performance, error boundaries, seguridad client-side
  • Complementa a senior-code-auditor que se enfoca mas en backend
📦 QUE GENERA
FRONTEND_AUDIT_REPORT.md — Tabla de puntuacion general (A-F por categoria), hallazgos por severidad (critico/alto/medio/bajo), detalle por categoria con archivo:linea y correccion, plan de accion recomendado.
💡 TIPS Y ERRORES COMUNES

✅ Ejecutala despues de senior-code-auditor para cobertura full-stack

✅ Verifica: "use client" innecesarios, falta de error.tsx/loading.tsx, next/image vs img

❌ No confundas con code-auditor — este profundiza en Next.js especificamente

❌ No asumas que todo Client Component es malo — verifica si necesita interactividad

📝 PROMPTS DE EJEMPLO
"Auditar el frontend del proyecto"
"Revisar componentes de Next.js — server vs client, performance, errors"
security-hardening-auditor
Auditor▼
📌 QUE HACE

Actua como un pentester profesional reportando a su cliente. Audita 12 categorias de seguridad: transporte, bootstrap (main.ts), rate limiting, autenticacion, autorizacion, validacion I/O, cabeceras HTTP, base de datos, secretos, dependencias, cookies/sesiones, y proteccion CSRF + file uploads. Genera un informe APROBADO/NO APROBADO.

⏰ CUANDO USAR
  • Despues de implementar o modificar autenticacion/autorizacion
  • Antes de deploy a produccion — verificar postura de seguridad
  • Cuando se agregan endpoints que manejan datos sensibles
  • Despues de tocar cookies, sesiones, o file uploads
📦 QUE GENERA
SECURITY_AUDIT.md — Checklist ✅/❌ de 12 categorias: Helmet, CORS, ValidationPipe, ThrottlerModule, guards, bcrypt, tokens, secrets, npm audit, cookies httpOnly/secure/sameSite, CSRF, file uploads. Con pasos de remediacion y codigo.
💡 TIPS Y ERRORES COMUNES

✅ Ejecutala cuando ya tengas auth implementado — asi el checklist completo aplica

✅ Ahora incluye cookies (httpOnly, secure, sameSite) y validacion de file uploads

❌ No la uses sin auth — la mitad del checklist va a ser N/A

📝 PROMPTS DE EJEMPLO
"Auditar seguridad del proyecto"
"Security review completo antes de ir a producción"
prisma-schema-auditor
Auditor▼
📌 QUE HACE

Audita el schema de Prisma modelo por modelo: nomenclatura, indices, relaciones, campos de auditoria (createdAt, updatedAt), soft delete (isActive/deletedAt), seeding, y convenciones. Verifica que la base de datos este optimizada y siga las mejores practicas antes de migrar en produccion.

⏰ CUANDO USAR
  • ANTES de hacer prisma migrate deploy en produccion
  • Cuando agregas modelos o relaciones nuevas
  • Para verificar indices y performance de queries
📦 QUE GENERA
Informe modelo por modelo — Nomenclatura, indices faltantes, relaciones mal configuradas, campos de auditoria, soft delete, convenciones. Con correcciones de schema concretas.
💡 TIPS Y ERRORES COMUNES

⚠️ Usala ANTES de migrar — si ya migraste, los fixes requieren otra migracion

✅ Complementa con deploy-readiness para verificar que la DB esta lista para prod

📝 PROMPTS DE EJEMPLO
"Revisar esquema de Prisma"
"Auditar la base de datos antes de migrar a producción"
testing-enforcer
Auditor▼
📌 QUE HACE

Audita la estrategia de testing del proyecto: existencia de tests unitarios, de integracion y e2e, calidad de las pruebas existentes, cobertura contra objetivos, y configuracion de CI para correr tests automaticamente. No busca 100% coverage ciego — prioriza tests de logica critica y flujos de negocio.

⏰ CUANDO USAR
  • Cuando baja la cobertura de tests o hay regresiones
  • Despues de agregar un feature — verificar que tiene tests
  • Antes de deploy — confirmar que los tests cubren los flujos criticos
📦 QUE GENERA
Informe de testing — Brechas de cobertura por modulo, pruebas faltantes, calidad de tests existentes (mocks, assertions, edge cases), configuracion CI, recomendaciones priorizadas.
💡 TIPS Y ERRORES COMUNES

✅ Prioriza tests de logica de negocio — services, no controllers

❌ No esperes 100% cobertura — la skill prioriza tests de lógica critica

📝 PROMPTS DE EJEMPLO
"Revisar mis tests y decirme qué falta"
"Auditar la estrategia de testing del proyecto"

Enforcer▼

⏰ CUANDO USAR
  • Cuando estas cerca de salir a produccion
  • Pre-deploy final — el ultimo checkpoint
  • Despues de configurar CI/CD para verificar que todo esta conectado
📦 QUE GENERA
Informe go/no-go — Variables de entorno, CI/CD, hosting, Sentry, logging, build de produccion. Por categoria con estado y recomendaciones.
💡 TIPS Y ERRORES COMUNES

❌ No la ejecutes demasiado temprano — si no tenes CI/CD, la mitad va a ser "falta configurar"

✅ Usala como ultimo paso antes del deploy

📝 PROMPTS DE EJEMPLO
"¿Está listo para producción?"
"Verificar deploy readiness del proyecto"
project-setup-enforcer
Enforcer▼
📌 QUE HACE

Verifica y configura la capa de enforcement del proyecto: todo lo que previene errores ANTES del commit. Audita 7 herramientas: TypeScript strict mode, ESLint, Prettier, Husky + lint-staged (pre-commit hooks), commitlint (conventional commits), GitHub Actions CI, y PR templates. Para cada herramienta faltante, genera la configuracion completa lista para copiar.

⏰ CUANDO USAR
  • Al inicio de un proyecto nuevo — configurar enforcement desde el dia 1
  • Cuando recibes un proyecto sin pre-commit hooks ni linting configurado
  • Para verificar que la configuracion no se desconfiguro con el tiempo
📦 QUE GENERA
Informe de enforcement — Tabla con estado ✅/🟡/❌ por herramienta. Para cada faltante: por que importa + configuracion completa (package.json, archivos de config, hooks). Si aceptas, instala y configura automaticamente.
💡 TIPS Y ERRORES COMUNES

✅ Ejecutala una vez al inicio del proyecto — despues solo para verificar

✅ Si es un MVP, commitlint puede esperar — la skill es pragmatica

❌ No instala nada sin preguntar primero — muestra el informe y despues pregunta

📝 PROMPTS DE EJEMPLO
"Configurar enforcement del proyecto"
"Setup pre-commit hooks, linting y CI"

Preguntas Frecuentes

Puedo usar feature-builder sin PROJECT_BLUEPRINT.md?

Si. Explora directamente la estructura del proyecto existente.

Es obligatorio aprobar el blueprint antes de implementar?

Si. HARD GATE impide escribir codigo sin tu aprobacion del blueprint.

blueprint-compliance-auditor funciona con FEATURE_BLUEPRINT.md?

No. Solo con PROJECT_BLUEPRINT.md (del generator o del rescue).

Cuando usar context-aware-coder vs feature-builder?

context-aware-coder para cambios pequenos, QA fixes, agregar campos, ajustes menores. feature-builder para funcionalidad nueva completa que necesita discovery, diseno y blueprint.

project-setup-enforcer se ejecuta una vez o siempre?

Una vez al inicio del proyecto para configurar todo. Despues solo para verificar que no se desconfiguro.

frontend-nextjs-auditor reemplaza a senior-code-auditor?

No. code-auditor ve backend + estructura general. frontend-auditor profundiza especificamente en Next.js: RSC boundaries, error.tsx, loading.tsx, performance, middleware.

Las auditorias individuales o la maestra (compliance)?

Con blueprint → maestra (una pasada). Sin blueprint → individuales. Despues de un feature → individuales para profundidad.

Cuando usar brainstorming vs senior-feature-builder?

feature-builder tiene Fase 0 de brainstorming integrada. Usa la skill brainstorming separada solo para explorar ideas que no son features (ej: decisiones de arquitectura, eleccion de libreria).

Cuantas skills hay en total?

19 skills: 1 generator, 2 builders, 7 auditors, 2 enforcers, 1 rescue, 6 reference/support.
